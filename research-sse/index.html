Server-Sent Events (SSE)

一种简单、强大的通过 HTTP 进行实时更新流传输的方式。

什么是 SSE?

SSE 是一种允许服务器通过 HTTP 向网页推送数据的标准。与 WebSockets 不同，它是单向的（服务器 → 客户端）。

主要优势:

  • 基于 HTTP: 无需特殊协议或服务器。兼容大多数代理和防火墙。
  • 自动重连: 如果连接中断，浏览器会自动尝试重新连接。
  • 轻量级: 与轮询或全双工 WebSockets 相比，开销较低。
工作原理: 服务器发送 Content-Type: text/event-stream 响应头并保持连接开启。

HTTP 响应头:

Content-Type: text/event-stream必须 告诉浏览器把响应 body 当作 SSE 事件流解析。没有它，EventSource 不会按 SSE 协议处理响应。
Cache-Control: no-cache强烈建议 SSE 是实时数据流，不应该被浏览器、代理或 CDN 缓存。缺少它时，某些中间层可能缓存、缓冲或延迟事件。
Connection: keep-aliveHTTP/1.1 场景 表达保持连接的意图，但不是 SSE 协议要求。只要服务端不结束响应，连接就可以持续；周期性发送注释行更能防止空闲超时。
Access-Control-Allow-Origin跨域时需要 允许不同源页面连接这个事件流。同源页面访问 /events 时不需要它，生产环境通常应限制为可信域名。

协议格式:

id: 123
event: notification
data: Hello World!
retry: 5000

: 这是一条注释
data: 事件负载。多个 data: 行会被浏览器用换行符合并成一次事件的 event.data。
event: 自定义事件名。省略时触发默认 message；设置后用 addEventListener() 监听对应名称。
id: 事件 ID。浏览器会记住最近的 ID，断线重连时通过 Last-Event-ID 请求头发回服务器。
retry: 设置自动重连等待时间，单位毫秒。它不是业务消息，而是更新 EventSource 的重连策略。
空行很重要: SSE 用一个空行结束当前事件。没有结尾的 \n\n，浏览器会继续缓冲，不会把事件派发给 JavaScript。

JavaScript API:

const source = new EventSource('/events');

source.onmessage = (e) => {
  console.log('新消息:', e.data);
};

source.addEventListener('notification', (e) => {
  console.log('自定义事件:', e.data);
});

EventSource 做了什么?

1. 打开连接 浏览器发起一个普通的 HTTP GET 请求到 /events。
2. 解析事件流 按 SSE 规范读取 data、event、id、retry 字段。
3. 自动恢复 连接失败后自动重连，并在可用时带上 Last-Event-ID。
